## 1 — Popup Error Fixes (completing 06-07 §12, which was never actually implemented)

Audit of `semantica-tauri/src/popup.js` and `semantica-tauri/src-tauri/src/lib.rs` found that Log_06-07.md §12 described three fixes (token sanitization, human-readable network errors, dev-build auto-DevTools) as done — none of them existed in the actual code.

- **Token sanitization:** added `sanitizeToken()` (trim + strip non-printable-ASCII) before the token is used to build the `Authorization` header. Prevents WebKit's `fetch()` from throwing "The string did not match the expected pattern" if the stored token ever picks up stray whitespace or non-ASCII bytes.
- **Human-readable network errors:** `TypeError` from `fetch()` (network failure / malformed request) now shows "Could not reach the Semantica server…" instead of the raw browser error; application-level errors (from the server's own `error` field) are shown as before. Also logged to console either way.
- Applied to both `semantica-tauri/src/popup.js` and `desktop/renderer/popup.js` (kept in sync, as they were identical).
- **Auto-open DevTools in debug builds:** implemented, then reverted. Opening DevTools moves OS focus to the inspector, which fires `WindowEvent::Focused(false)` on the popup window — this trips the *existing* blur-dismiss handler and closes the popup instantly. Manual right-click → Inspect Element works without this side effect, so the auto-open was removed from `lib.rs`.

## 2 — KV Resilience

Cloudflare Workers KV enforces two hard limits: max 1 write/second to the *same* key (both plans), and on the free plan, max 1,000 writes/day to *different* keys account-wide. The IPA KV bulk upload (65,119 keys, still pending from 06-07 §8) would exhaust the free-tier daily quota in a single command if run on a free account, breaking every other KV write for the rest of that day.

- `server/src/rate-limit.js`: `checkRateLimit()` now fails **open** (allows the request, logs the failure) instead of throwing on a KV `get`/`put` error. A rate-limit bookkeeping failure no longer 500s the whole `/v1/analyze` or `/v1/translate` request.
- `server/src/analyze.js`: the analysis cache read and write are both wrapped in try/catch with logging — a KV hiccup degrades to "treat as cache miss" / "skip caching this result" rather than crashing an otherwise-successful analysis.
- Added `console.error` logging (status + response body snippet) throughout `_callDeepLTexts` and `_claudeTranslateFallback`, which previously swallowed every failure into a silent `null` with no way to diagnose it via `wrangler tail`.

## 4 — Two-Step Analyze/Translate Split

**Goal:** show English content (word, POS, phonetic, definition, example) immediately instead of waiting for the DeepL translation too, then patch the Vietnamese translation in when it's ready.

- `server/src/analyze.js` refactored into two shared helpers:
  - `_getOrBuildEnglishContent(text, context, env)` — Phase 1 (Claude + Google TTS + Datamuse + IPA), cached under a new **language-independent** key `wac_en_{word}` since Claude's English output never varies by target language.
  - `_applyTranslation(analysis, text, targetLanguage, env)` — Phase 2 (DeepL + Claude fallback), unchanged translation logic, now reusable.
  - `handleAnalyze` (existing `/v1/analyze`, still used by the dashboard's lookup modal) now composes these two helpers internally — contract and behavior unchanged.
- **New endpoints**, wired into `server/src/index.js`:
  - `POST /v1/analyze/fast` — Phase 1 only. Checks the full per-language cache first; returns `{ ...cached, complete: true }` on a hit, otherwise English content with `complete: false`.
  - `POST /v1/analyze/translate` — Phase 2 only. Rebuilds English content from cache if needed (so it's safe to call standalone), translates, persists the merged full result to the per-language cache, returns just the translation delta fields (`translation`, `definitionTranslated`, `exampleSentenceTranslated`).
  - Both reuse the existing `analyze` / `translate` rate-limit buckets — no new quotas added.
- `semantica-tauri/src/popup.js` (⚠️ **not ported to `desktop/renderer`** — legacy Electron app intentionally left alone for this feature, per the project's own "kept intact" note): calls `/v1/analyze/fast` first and renders immediately; if `complete: false`, calls `/v1/analyze/translate` in the background and patches the result into a stable `#meaning-slot` DOM node via a new `patchTranslation()` function, with a small shimmer placeholder shown in that slot while pending.
- Added a request-sequence counter (`_requestSeq`) in `popup.js` — the popup window is reused across searches, so a stale response from an earlier word (especially the second leg of the two-step chain) is now detected and dropped instead of overwriting a newer word's card.

## Known Issue — End of Session (not yet fixed)

**Symptom:** popup shows English content (word, POS, phonetic, definition, example) correctly, but the translation section never appears — no shimmer, no Vietnamese text, and it doesn't resolve no matter how long the user waits.

**Investigation so far:** the content shown for a repro case ("loading") is a byte-for-byte match to the offline `local-dictionary.json` entry for "load" (definition, phonetic, and example sentence all identical) — meaning the render came from the **offline dictionary rescue path**, not a real server response. That path was already designed to be English-only with no translation attempt (pre-existing behavior), and critically, the code path it falls into after a fetch failure (`renderResult(word, _currentAnalysis, false)`) defaults `pendingTranslation` to `false`, so no shimmer appears and no retry ever happens.

**Leading hypothesis:** the `/v1/analyze/fast` request is failing outright. Two candidate causes, not yet confirmed:
1. **Not deployed** — `popup.js`'s `BACKEND_URL` is hardcoded to the production Worker URL; running `wrangler dev` locally does *not* make the popup talk to it. If `npx wrangler deploy` hasn't been run since `/v1/analyze/fast` and `/v1/analyze/translate` were added this session, every call to them 404s on production, which is caught and falls through to the offline dictionary rescue.
2. **A real, separate cache-poisoning bug**, found during this investigation regardless of the above: `handleAnalyzeTranslate` (and originally `handleAnalyze`, unchanged from before this session) unconditionally writes the merged analysis to the full per-language cache even when both DeepL and the Claude translation fallback fail — caching empty translation fields as if the entry were "complete" for a full 7-day TTL. Once a word is poisoned this way, `/v1/analyze/fast` reports `complete: true` with empty translation forever, and the client never attempts step 2 again.

**Resolved:** root cause was cause 1 — the two-step endpoints weren't deployed yet. User ran `npx wrangler deploy`; two-step fast/translate flow confirmed working after redeploy. Cause 2 (cache-poisoning bug) was not hit in practice but is still present in the code — see Pending / Next Steps below.

## 5 — Dashboard & Popup Follow-ups (post-deploy)

**Capture words lowercase.** Both places a word gets written into the vocabulary now lowercase `text` at the point of saving:
- `semantica-tauri/src/popup.js` `handleSave()` — `text: (_currentAnalysis.baseForm || _currentWord).toLowerCase()`.
- `semantica-tauri/src/app.js` lookup-modal save handler — `saveWord({ text: _pendingWord.toLowerCase(), ... })`.

Only the *stored* text is lowercased — the word passed to the analyze API call and shown while looking it up is untouched, so analysis quality/display isn't affected. This makes casing typos/inconsistencies stop creating duplicate-looking vocab entries (the duplicate checks already compared lowercase, but the stored value itself wasn't normalized).

- If the new text collides with another existing word (case-insensitive), an inline "Already in your vocabulary" warning shows for 1.4s and then reverts without saving, rather than silently creating a duplicate.
- A `settled` flag prevents the blur handler from re-committing after Enter/Escape/duplicate already resolved the edit (blur still fires when the input is removed from the DOM).

**Reload button removed.** The "↻ Reload" button (re-runs `/v1/analyze` for a single saved word and overwrites its `aiAnalysis`) is gone from the Actions column in `renderWords()`. ## 6 — Fix Vocab Search State Lost on Tab Switch

**Symptom:** typing a search term in "My Vocabulary," then switching to another tab (e.g. Game) and back, left the search box showing the typed text but the list below reverted to the full unfiltered vocabulary.

**Root cause:** `app.js`'s tab-click handler re-rendered the list on return to `list-view` via a bare `loadVocab().then(renderWords)` — no filter argument, no awareness of the search box's current value. The actual filtering logic, `applySearchFilter(term)`, was defined as a function *nested inside* `initVocabSearch()`, so it wasn't reachable from the tab-click handler.

**Fix:**
- Hoisted `applySearchFilter(term)` out of `initVocabSearch()` to top-level scope. It now re-queries its own DOM refs (`search-define-prompt`, `search-define-label`, `vocab-search-input`) internally instead of closing over `initVocabSearch()`'s locals, so it's safely callable from anywhere in the file (same hoisting pattern already relied on elsewhere in `app.js`).
- `initVocabSearch()`'s own `input` listener still calls `applySearchFilter(_searchTerm)` as before — now resolving to the hoisted version.
- Tab-click handler's `list-view` branch changed from `loadVocab().then(renderWords)` to re-reading the search input's current value and calling `applySearchFilter(value)` after `loadVocab()` resolves. An empty value already falls back to the full unfiltered list inside `applySearchFilter`, so this is a strict superset of the old behavior — no regression when the search box is empty.

Verified with `node --check app.js` (syntax OK) and confirmed exactly one `applySearchFilter` definition remains (the hoisted one).

## 10 — Windows: Blank Translate Popup After Tray Search

**Symptom (first real Windows test, post-deploy):** using the tray icon to search and translate a word opens the popup window, but its content never renders — permanently blank white, no error, no recovery. The quick-search bar itself was unaffected (only the translate popup). A follow-on report: after this happened once, a full computer shutdown (not a clean app quit) while the popup was stuck left `com.mysemantica.app`'s WebView2 profile (`%LOCALAPPDATA%\com.mysemantica.app\EBWebView`) corrupted, and the app failed to launch at all afterward (no window, no tray icon) until that folder was deleted.

**Diagnosis (code inspection + elimination, no live Windows console access — devtools weren't compiled into this build):**
- Ruled out a systemic WebView2/runtime problem: the main dashboard window rendered fully and correctly, so WebView2 itself was working.
- Ruled out a generic windowing/transparency quirk: the popup and the search bar are built through the *identical* `WebviewWindowBuilder` chain (`decorations(false)`, `transparent(true)`, `always_on_top(true)`, `resizable(false)`, `skip_taskbar(true)`, `shadow(true)`, `focused(true)`) — if that config were the problem, search would fail too, and it didn't.
- What's actually unique to the popup: `search_word` (`lib.rs`) closes the search window and creates the popup window back-to-back in the same synchronous command (`close_search(app.clone()); show_translate_popup(&app, &word, saved);`), and the popup depends on an async Rust→JS data handoff (`pending_popup` Mutex + `popup_ready` invoke / `analyze-word` event) that the search bar doesn't need at all. Windows shares a WebView2 environment across an app's windows; destroying one webview and immediately creating another in the same tick is a known source of silent init failures on Windows specifically (WKWebView on macOS doesn't share this fragility) — and every failure path in this chain was silently discarded (`let _ = ...build()` in Rust, `.catch(() => {})` in `popup.js`), so a failure here would produce exactly what was reported: a window that exists and paints its background, never receives content, and never errors.
- The subsequent "won't launch after a full shutdown" was a downstream consequence, not a separate bug: the popup was stuck in a half-initialized webview state at the moment of the abrupt kill, which is exactly the condition known to corrupt a WebView2 profile folder.

**Fix (`semantica-tauri/src-tauri/src/lib.rs`, `popup.js`, `Cargo.toml`):**
- `search_word`: added a 150ms `std::thread::sleep` between `close_search()` and `show_translate_popup()` to give WebView2 room to tear down the old webview before a new one is created. Runs off the main thread (Tauri already dispatches sync commands to a worker pool), so this doesn't block the UI.
- `show_translate_popup` and `show_word_of_day`: `WebviewWindowBuilder::...build()`'s `Result` is no longer discarded — a failure now logs via `eprintln!` instead of vanishing silently.
- `popup.js`: added a 5-second safety-net timeout that fires only if the popup has *never* received any word data since loading — shows a visible "couldn't load this word, close and try again" error instead of leaving the window blank forever. Does not affect normal operation (cleared by the first successful dispatch).
- `Cargo.toml`: added the `devtools` Cargo feature so right-click → Inspect Element works in this build going forward — the previous absence of this feature is why there was "no way to access the dev view" while debugging this. Per Tauri's own guidance this isn't meant to ship long-term (it enables private macOS APIs that block App Store acceptance) — fine for now since distribution is direct-download via GitHub Releases, not App Store, but worth revisiting before any future store submission.

**Status:** implemented, not yet verified on Windows — needs a rebuild + redeploy (GitHub Actions) or a local Windows build, then a re-test of tray → search → translate. If the popup still goes blank after this, DevTools is now available to get an actual console error instead of debugging blind.

### 10a — Follow-up: the 150ms sleep didn't fix it; real root cause found and fixed

**New evidence:** a screenshot of the still-blank popup on a Windows build made *after* the 10 fix shipped. The window is titled **"Tauri App"** with full native title-bar chrome (minimize/maximize/close) — but `show_translate_popup` explicitly sets `.decorations(false)` and `.resizable(false)` and never calls `.title(...)`. That mismatch means the window's builder configuration wasn't applied at all — it fell back to raw platform defaults, not just "blank content."

**Actual root cause:** confirmed directly from the installed `tauri` crate's own source (`tauri-2.11.5/src/webview/webview_window.rs:56-59`, `113-116`):

> On Windows, this function deadlocks when used in a synchronous command and event handlers, see [the Webview2 issue] (`tauri-apps/wry#583`). You should use `async` commands and separate threads when creating windows.

`search_word` and `show_word_of_day` (`lib.rs`) are both non-`async fn` `#[tauri::command]`s that call `WebviewWindowBuilder::build()` directly — exactly the documented failure case. This also explains the "Tauri App" title: `tauri-utils-2.9.3/src/config.rs`'s `default_title()` literally returns `"Tauri App"`, the fallback used when a window's builder config never finishes applying.

The 10 fix's 150ms sleep was based on a different, incorrect theory (a WebView2 *teardown-timing* race) — it doesn't touch the actual constraint (which thread/dispatch context `build()` runs in), so it left the bug in place. The screenshot is direct evidence of that.

**Also found:** `main.rs` sets `#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]`, which detaches stderr in release builds (i.e. the GitHub Actions build via `tauri-action` in `.github/workflows/tauri-release.yml`). The `eprintln!` error logging added in the 10 fix is therefore silently discarded in exactly the build being tested — it would never have surfaced a `build()` failure even if one occurred cleanly instead of degrading to defaults.

**Fix (`semantica-tauri/src-tauri/src/lib.rs`):**
- `search_word` and `show_word_of_day`: changed from `fn` to `async fn` (returning `Result<(), ()>`, required by Tauri's command macro once a reference-typed extractor like `tauri::State` is combined with `async`). This routes both commands through Tauri's async dispatch instead of the sync-command worker pool, per the docs' own recommended fix. Removed the now-superseded 150ms `std::thread::sleep` from `search_word` — it targeted the wrong mechanism and is no longer needed.
- Added `log_window_error()`: mirrors `WebviewWindowBuilder::build()` failures to a `window-errors.log` file in the app's log dir (`app.path().app_log_dir()`) in addition to `eprintln!`, so a failure in a packaged Windows build leaves evidence instead of vanishing into a detached stderr. Wired into both `show_translate_popup`'s and `show_word_of_day`'s `build_result` error branches.
- No JS changes needed — `tauri-bridge.js`'s `invoke()` calls are promise-based regardless of whether the Rust-side command is sync or async.

Verified with `cargo check` (clean) and `cargo clippy` (one pre-existing, unrelated warning at a different line; nothing new introduced).

**Status:** implemented, not yet verified on Windows — needs a rebuild + redeploy, then a re-test of tray → search → translate. If it still fails, check `window-errors.log` in the app's log dir first.

### 10b — Separate bug: main window never shows once already logged in

**Symptom:** first launch works fine (login page appears, sign-in works, popup works). After closing and reopening the app, nothing shows — no dashboard window, and this is not the WebView2 popup bug in §10/§10a; it's the *main* window that fails to appear this time.

**Root cause (found by code inspection, not platform-specific — applies to every OS the Tauri build runs on):** `tauri.conf.json` starts the `"main"` window with `"visible": false`. The only thing that ever calls `.show()` on it is `window.electronAPI.showDashboard()` → the Rust `show_dashboard` command. In `semantica-tauri/src/app.js`'s `DOMContentLoaded` handler, that call was gated inside the *logged-out* branch only:

```js
if (isLoggedIn()) {
  showMainApp();
} else {
  showLoginPage();
  window.electronAPI?.showDashboard?.();   // ← only path that ever reveals the window
}
```

`showMainApp()`/`showLoginPage()` only toggle which `<div>` is visible *inside* the page — they have no effect on the native window's visibility, which is a separate concern entirely. On first run the user isn't logged in yet, so the `else` branch runs, `showDashboard()` fires, and the (still-hidden) window becomes visible for the first time — login page appears, user signs in, everything works. `cloud_auth_token` then persists in the webview's localStorage across restarts (same profile dir), so on every subsequent launch `isLoggedIn()` is true, the `if` branch runs, and `showDashboard()` is never called — the window stays permanently hidden with nothing left to reveal it. The app process itself is still running fine (tray icon present, commands still work) — it just never puts a window on screen.

This is a plain frontend logic bug, not a WebView2/Windows quirk, so it should reproduce on macOS too on a genuinely cold launch (process fully quit, not just window-hidden) while already logged in — macOS testing likely didn't catch it because clicking the Dock icon while the process is still alive in the background fires `RunEvent::Reopen` (`lib.rs`, `#[cfg(target_os = "macos")]`-gated), which does call `.show()` and papers over the bug in that specific interaction. Windows has no equivalent fallback anywhere in the code, so it's the platform where this was actually noticed.

**Fix (`semantica-tauri/src/app.js`):** moved the `showDashboard()` call out of the `else` branch so it runs unconditionally after either `showMainApp()` or `showLoginPage()`, regardless of auth state:

```js
if (isLoggedIn()) {
  showMainApp();
} else {
  showLoginPage();
}
window.electronAPI?.showDashboard?.();
```

No-op on Electron either way (`desktop/renderer/app.js` has the identical gated pattern but is unaffected in practice — the legacy Electron main window is never hidden in the first place, so `showDashboard()` being skipped there was always harmless). Left untouched, consistent with this session's "legacy, kept intact" scoping.

Verified with `node --check app.js` (syntax OK).

**Status:** implemented, not yet verified on Windows — needs a rebuild + redeploy, then a re-test of: sign in → fully quit the app (not just close the window) → relaunch → dashboard should appear immediately.

## §15 — Revert Phaser Race mode integration (blank canvas on rebuild)

User rebuilt and tested the Road Trip scene — the canvas area rendered as a completely empty rounded box (visible only because `#race-canvas-container`'s own CSS background color showed): no sky, no mountains, no car, nothing. Asked to identify the error and revert everything, with a "discuss before committing again" note.

**Diagnosis (from static review only — no console/devtools access in this sandbox, so this is the best-supported hypothesis, not a confirmed root cause):** the empty box having its own background color visible confirms `#race-canvas-container` was correctly set to `display:block`, which only happens when `isRaceSceneAvailable()` returns true — meaning the Phaser library itself loaded fine from the CDN. The failure is downstream of that: nothing from `create()` ever drew (not even the sky gradient, which was the very first thing drawn), which points to `create()` never running at all. The strongest candidate: `preload()`'s `this.load.svg('car', dataUri, ...)` call, where `dataUri` was a raw (non-base64) `data:image/svg+xml;...` URI. Phaser's SVG loader fetches the SVG source via XHR before rasterizing it, and WKWebView (what Tauri uses for the macOS build) has a long-standing restriction blocking `XMLHttpRequest` requests against `data:` URLs — if that fetch never resolves, Phaser's loader queue never completes, and `create()` is never called by the Scene lifecycle. A secondary, less certain contributor considered: `Phaser.Scale.FIT` computing its initial canvas-to-parent fit before the container has been laid out (if `mountRaceScene` ran in the same synchronous tick as the `showScreen()` class toggle, before the browser's next layout pass) could produce a 0-sized canvas. ## §26 — Fix: Practice Enter key collapsed both steps into one press

User reported §25's Practice Enter-to-advance wasn't behaving as a real two-step flow. Root cause, found by tracing the actual event dispatch rather than guessing: each mode's answer `<input>` had its own `keydown` → `checkAnswer()` listener (submit), and a separate document-level listener (added in §25) that clicks Next once it's visible. Both are attached along the same DOM path (`input` → ... → `document`), and a single `keydown` event bubbles through *all* of them in one synchronous dispatch. On the very first Enter press: the input's listener fires first (target phase) and calls `checkAnswer()`, which synchronously reveals the answer *and* makes the Next button visible — then the *same* event keeps bubbling up to `document`, where the Enter-to-advance listener now sees a visible Next button and immediately clicks it. Net effect: one Enter press both submitted the answer and advanced past the results screen in the same instant — the reveal was never actually visible, exactly what the user meant by "accidentally skip questions without answering them" (functionally, it looked like Enter did nothing except immediately move on).

**Fix:** added `e.stopPropagation()` to both modes' input `keydown` handlers, so the "submit" keypress never reaches the document-level "advance" listener in the same dispatch. State 1 (typing → Enter) and State 2 (results showing → Enter) are now guaranteed to be two separate keypresses, because the input is disabled (and loses focus) the instant `checkAnswer()` reveals the answer, so there is no longer any single event that both handlers can see.

**Second fix, required to make "State 2 (after submitted *or* skipped)" true:** Skip previously didn't go through any reveal step at all — `skipBtn.onclick` just did `_practiceIndex++; _showNextFitb()` immediately, silently jumping to the next word with no feedback shown. That meant there was no "results screen" for Enter to advance *from* after a skip, contradicting the spec's explicit "after they submitted an answer or clicked skip" wording. Extracted the shared show/hide-button transition (input disabled, Check/Skip/Hint hidden, Next shown) into a small `_revealUI()` helper used by both `checkAnswer()` and a new `skipAnswer()` function; skip now reveals the correct word + definition (styled as a neutral "→ Skipped" message, not counted in `_practiceCorrect`/`_practiceWrong` — same neutral semantics Skip already has everywhere else in the app) and waits for Next/Enter like a real answer does, instead of auto-advancing.

Per the explicit instruction to keep Skip itself visible: Skip remains a normal, always-available button during State 1 (answering) in both modes — nothing about this fix removes or hides it before use; it only now hides *after* being clicked, the same way Check/Hint already do once their action is taken.

One more edge case closed while in there: if a user tabs to focus the Next button directly and presses Enter, the browser's own native "Enter activates a focused button" behavior fires its own click *in addition to* the document-level listener's explicit `nextBtn.click()` — without a guard this would advance two words per keypress. Added `if (document.activeElement === nextBtn) return;` to the document-level listener so it defers to the browser's native click in that specific case instead of double-firing.

`game-modes.js` and every other file untouched — this fix is scoped entirely to the two `_showNextFitb()`/`_showNextListen()` functions in `app.js`. Verified: `node --input-type=module --check` on `app.js`; grepped to confirm both `skipBtn.onclick` sites now point to `skipAnswer` (not the old inline auto-advance arrow) and both `nextBtn.onclick` sites are untouched (still the only place `_practiceIndex++`/`_listenIndex++` happens). **Not visually tested** — no browser in this sandbox; worth manually running through both Practice modes end-to-end (type-and-Enter, Check-button-click, Skip-button-click, and Tab-to-Next-and-Enter) to confirm each path now lands on the results screen exactly once before advancing.
