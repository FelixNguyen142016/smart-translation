## 2 — Server: cache-poisoning fix + Vietnamese default (`server/src/analyze.js`)

- New `_cacheFullResult(env, text, targetLanguage, analysis)` helper: skips the per-language full-cache write when `analysis.translation` is empty (both DeepL and the Claude fallback failed), so failed translations are retried on the next lookup instead of being cached as "complete" for 7 days. Used by both `handleAnalyze` and `handleAnalyzeTranslate`.
- `handleAnalyzeFast` no longer trusts a cached full entry without a `translation` — pre-fix poisoned entries self-heal: the English content is served with `complete: false`, the client retries the translate step, and a successful result overwrites the poisoned entry.
- `targetLanguage` default changed `'Spanish'` → `'Vietnamese'` in `handleAnalyze`, `handleAnalyzeFast`, `handleAnalyzeTranslate`.

## 3 — Server: `POST /v1/vocab/word` (single-record append)

New endpoint in `vocab.js` (routed in `index.js`): appends one word record server-side in a single request — D1 read, case-insensitive duplicate check against both `text` and `aiAnalysis.baseForm`, unshift, write. Returns `{ ok, added }`. `popup.js` `handleSave()` now uses it instead of GET-full-list → mutate → PUT-full-list, which (a) closes the lost-update race where a popup save could clobber a concurrent dashboard write with a stale copy, and (b) shrinks the request from the entire vocabulary to one record. The dashboard's full-array PUT (`api.js`) is unchanged — it legitimately manages the whole list.

## 4 — Tauri: popup-reuse race fix (`lib.rs` + `tauri-bridge.js`)

Previously: first popup open used the `pending_popup` stash + `popup_ready` pull, but *reuse* sent the payload via `emit_to("popup", "analyze-word", payload)` — lost forever if the window existed while its JS hadn't registered the listener yet (fast consecutive searches). Now Rust **always** stashes the payload in `pending_popup` and emits a bare `popup-wake` event; the payload never rides on an event. The bridge pulls via `popup_ready` on every wake **and** once right after registering the wake listener (`listen('popup-wake', pull).then(pull)`), so no request can fall into the registration gap. Double-pulls are safe (`take()` → `None` → ignored); newer requests overwrite older pending ones, so the newest word always wins. Applied to both `show_translate_popup` and `show_word_of_day`.

## 11 — Cost & Infrastructure Hardening Roadmap (planning session, nothing implemented yet)

Fourth work block this session — a cost audit and an NFR (non-functional requirements) audit of `server/`, done in conversation, not yet turned into code. Full cost numbers, provider comparisons, and pricing math live in `Cost-Billing-Analysis.md`; this section is the engineering-task summary.

**Cache TTL (the highest-leverage single change identified):** `CACHE_TTL_SECONDS` in `analyze.js` is currently 7 days. Since word definitions/translations don't actually go stale, the 7-day expiry causes popular words to be re-translated (re-billed to DeepL/Claude/TTS) roughly weekly across the whole user base for no real reason — modeled at roughly 50-60% of total API cost at every scale tested. Extending to 1 year, combined with pre-seeding the cache from a teacher-curated, GPT-4o-mini-generated word list (funded by the existing $5,000 token.ai.vn credit — that credit's runway at this usage is measured in decades, not months, so it's treated as free), models out to an 80-90% total cost reduction versus the current 7-day baseline. The two changes only work together: pre-seeding without the longer TTL evaporates within a week and has to be re-run constantly to have any effect.

**NFR audit of `server/src`** (Performance / Scalability / Security / Availability / Reliability / Rate Limiting / Logging & Auditability / Error Handling / Maintainability), verified against actual code, not assumed:

- **Already solid:** error handling (global `onError` + consistent `errResponse()` shape + correct status codes across all handlers), rate limiting (real per-user/per-day KV-backed limits, enforced *before* the expensive upstream call), maintainability (clean single-purpose modules, consistent `/v1/*` versioning), and webhook idempotency (SePay dedup by `sepay_tx_id` with a DB unique index).
- **Gaps found:** no timeouts on any upstream fetch (Claude/DeepL/TTS/Datamuse can all hang the Worker indefinitely), no retry/backoff anywhere, bearer tokens never expire once issued, input validation is presence/type-only with no payload size caps, and logging is unstructured `console.log`/`console.error` with no correlation ID — tracing one request across three upstream providers isn't currently possible from logs alone. No tests exist anywhere in `server/`.

**Staging by scale:** none of the gaps above require a redesign — the API's shape (stateless, versioned, cleanly separated) already holds up. The operational-hardening items (timeouts, retries, structured logging, token expiry, payload caps, moving off the Workers free plan, confirming upstream provider rate-limit headroom) are the "do before real growth, cheap to do now" tier — realistic to complete well before 10,000 users. D1 read-replica scaling, dedicated APM tooling, and RBAC are explicitly deferred — none of them are load-bearing at 10K users based on the current cost model (Cloudflare infra cost stayed around $5-10/month even at the 10K-user tier in every scenario modeled), and building them early would be solving a problem that doesn't exist yet.

## 13 — Post-mortem: CI release builds failing with "A public key has been found, but no private key" (RESOLVED)

Multiple tag pushes (v1.3.0 → v1.6.1) failed with this error even after the signing secrets were set and re-set several times. Recorded here because the failure mode is misleading and worth remembering.

**Root cause: a stale duplicate workflow on GitHub, not the secrets.** The retirement of the legacy `build-tauri.yml` (moved to `.github/workflows-retired/` during the fix round) had only ever happened in the local working tree — it was **never committed and pushed**, so GitHub still had it. Every `v*` tag therefore triggered **two** parallel builds: `tauri-release.yml` (tauri-action@v1, signing env wired — healthy) and the stale `build-tauri.yml` (tauri-action@v0, **no signing env vars at all**). With `createUpdaterArtifacts: true` in `tauri.conf.json`, any build without `TAURI_SIGNING_PRIVATE_KEY` in its environment fails with exactly this error — so the legacy workflow failed on every tag no matter how correctly the secrets were configured. Rewriting the key could never fix it; the failing job never asked for the secret.

**Contributing confusions ruled out along the way:**
- Tag naming (jumping to v1.3.0 instead of v1.2.0) — irrelevant; the tag name never affects signing, it only triggers the workflow.
- Secret placement — verified correct via screenshot: all three (`TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`, `RELEASES_REPO_TOKEN`) existed as **Repository secrets** in `smart-translation` with exact names.
- Key file format — `~/.tauri/semantica.key` is one long base64 string (decodes to "untrusted comment: …"; Tauri's normal format). The terminal merely wraps it visually; the safe copy method is `pbcopy < ~/.tauri/semantica.key` so no newlines sneak into the pasted secret.

**Fixes applied:**
1. Added a **fail-fast guard step** to `tauri-release.yml` ("Verify signing secrets are configured") that errors in the first seconds with a plain-language message if `secrets.TAURI_SIGNING_PRIVATE_KEY` or `secrets.RELEASES_REPO_TOKEN` expands empty — GitHub silently substitutes `""` for missing secrets, which otherwise surfaces 20 minutes into a build as Tauri's confusing error.
2. Committed + pushed everything (commit `b18e9fe`), which finally deleted `build-tauri.yml`/`build.yml` from GitHub — one release pipeline remains.
3. Re-saved the private-key secret via `pbcopy` (single line guaranteed) and confirmed the `tauri.conf.json` pubkey matches the current (rotated 20-07) `semantica.key.pub`.

**Outcome: v1.6.2 built, signed, and published successfully** to `semantica-releases` — the auto-update pipeline is live end-to-end. Users on v1.6.2+ receive future releases in-app.

**Lessons for future notice:**
- `${{ secrets.X }}` expands to an **empty string with no warning** when the secret doesn't exist/isn't visible — guard steps that check for empties are worth the 5 lines in every workflow that needs secrets.
- Workflow changes only take effect once **pushed** — retiring/renaming a workflow locally does nothing to what GitHub runs on the next tag.
- When a tag triggers multiple runs, check **which workflow file** the failing log belongs to before debugging its inputs (the `@v0` vs `@v1` in the log header was the giveaway here).
- The updater signing keypair must **never rotate again** now that v1.6.2 has shipped — installed apps verify against the baked-in pubkey; a rotated key means every existing install rejects all future updates.

## 14 — Popup broken: "Could not fetch definition" (token.ai.vn integration, IN PROGRESS)

Sixth work block this session. User reported the popup showing:

> Could not fetch definition. Analysis failed: Unexpected token '<', "<!doctype "... is not valid JSON

**Diagnosis so far:**

1. **Wrong base URL path (fixed).** `TOKENAI_BASE_URL` in `server/wrangler.toml` was `https://token.ai.vn` (no path). `_callGPTDefinition` (`analyze.js`) appends `/chat/completions`, hitting `https://token.ai.vn/chat/completions` — a path that doesn't exist on that host. token.ai.vn falls back to serving its normal HTML page with a `200 OK` (common catch-all routing) instead of a 404, so `res.ok` was true and `res.json()` choked parsing `<!doctype html>...` as JSON — exactly the error the user saw. Confirmed via curl test against both paths: `/v1/chat/completions` returned real structured JSON (`{"error":{...}}`), the no-prefix path returned the same HTML shell as the popup error. token.ai.vn runs **"One Hub"** (a self-hosted OpenAI-proxy/gateway), which uses the standard OpenAI `/v1/...` path convention.
   - **Fix applied:** `TOKENAI_BASE_URL` → `https://token.ai.vn/v1` in `wrangler.toml` (code still appends `/chat/completions`, now correctly forming `/v1/chat/completions`).
2. **Defensive error handling added regardless** (`analyze.js`, `_callGPTDefinition`): a `200` response is no longer trusted blindly — checks `content-type` for `application/json` and catches JSON parse failures explicitly, throwing a clear diagnostic error (which base URL was used, what content-type came back, first 200 chars of the body) instead of letting a raw `SyntaxError` bubble up as a cryptic message. This makes any future base-URL misconfiguration fail loudly and legibly rather than reproducing this exact confusing symptom.
3. **New blocker found via curl: `401` "无效的令牌" (invalid token) on the corrected `/v1/chat/completions` path**, confirmed with the user's real API key (tested twice). Since the same key works from a separate tool the user has configured against the same host, this isn't a bad/mistyped key — most likely the token is scoped/restricted in the One Hub dashboard (group/channel binding, or an IP/origin restriction limiting it to requests from that other tool). This matches a note the user had saved about the key needing to be set up as a "code AI provider" — terminology specific to token.ai.vn's own dashboard that isn't independently documented anywhere findable via search.

**Status: UNRESOLVED.** Waiting on the user to check the token's settings in the One Hub dashboard (group/scope/provider binding, IP or origin restrictions) and either confirm/loosen those, or generate a fresh unrestricted token for the Worker to use. Once a working key + confirmed path combination is in hand, remaining steps are unchanged from §12's checklist: `npx wrangler secret put TOKENAI_KEY`, `npx wrangler deploy`, then a real end-to-end lookup test.

**Follow-up (same day, after v1.6.2 shipped):** user reported the popup still showing the original raw `Unexpected token '<', "<!doctype"` error. Key realization: the current local code *cannot produce that message anymore* — `_callGPTDefinition`'s content-type guard rewrites it into a clear diagnostic — so the error proves **production is still running the pre-§14 Worker** (`npx wrangler deploy` was never run; the base-URL fix and diagnostics exist only locally). Deploying alone would only trade this error for the clearer 401, since the token itself is still rejected.

**Resilience fix applied (`analyze.js`):** the pipeline no longer hard-depends on token.ai.vn. New `_claudeDefinitionFallback()` — same `definitionSystemPrompt()`, run on Claude Haiku (max_tokens 400, the provider that handled definitions solo before the §12 split) — is invoked whenever `_callGPTDefinition` fails, with the GPT failure logged for diagnosis. Net effect: after the next `wrangler deploy`, word lookups work again immediately at Claude's (higher) cost, and token.ai.vn becomes a cost optimization to finish when the 401 is resolved — not a single point of failure. Verified: `node --check` on `analyze.js`.

Fix: model is now configurable via a new `TOKENAI_MODEL` var in `wrangler.toml`, set to `gpt-4.1-mini` (confirmed available); `_callGPTDefinition` reads it with a `gpt-4.1-mini` default.

## 15 — Popup "Save failed" on an end user's exe build (DIAGNOSED, not yet fixed)

Seventh work block this session. User reported "Save failed" in the popup on a real end user's machine (Windows `.exe` release build), while `cargo tauri dev` on the user's own Mac worked fine.

**Diagnosis:**

1. **Not a network/platform issue.** `popup.js`'s `handleSave()` catch block (line ~334) shows `"Could not reach the Semantica server."` for a `TypeError` (network/CORS/DNS failure) and `err.message` otherwise. The screenshot showed the literal text **"Save failed"**, which only comes from line 330 (`if (!postRes.ok) throw new Error('Save failed')`) — meaning the request reached the Worker and got a real HTTP error response back. This rules out a Windows-vs-macOS WebView networking difference as the cause, and rules out the user's recent UI edits (nothing in `popup.js` was touched this session).
2. **Real bug found: `popup.js` swallows the server's actual error.** Line 330 throws a hardcoded generic message instead of reading the response body, so the true reason (401? 500? something else?) is discarded before it ever reaches the UI. This needs fixing regardless of the root cause below, so failures like this stop being a guessing game.
3. **Most likely root cause: single-session auth design (`server/src/auth.js`).** `handleAuthVerify` (line 76-79) does `UPDATE users SET token_hash = ?` — one token column per user account, not a session table. Logging in on a second device **silently invalidates** the first device's token (its hash no longer matches any row, so `verifyToken` returns `null` on the next request). If the same account was logged into both the Mac dev build and the Windows exe, whichever logged in more recently silently kicked the other's session — the older session keeps *looking* fine until it hits an authenticated write like Save, which then 401s. This fits the reported symptom exactly (works on one device, fails with a bare "Save failed" on the other, no other visible breakage).

**Not yet confirmed:** whether this actually is a two-device same-account login collision — needs a real repro (log out + back in on the failing machine, retry Save) to verify before treating it as confirmed root cause rather than the leading hypothesis.

**Fixes needed (not yet implemented):**
- [ ] `popup.js` `handleSave()`: read the response body on `!postRes.ok` (status + `data.error`) instead of throwing a generic "Save failed", so the real server-side reason reaches the UI
- [ ] Confirm the single-session hypothesis: reproduce by logging out/in on the affected machine and retrying Save
- [ ] Product decision needed: is single-device-at-a-time login acceptable long-term, or should `auth.js` move to a real multi-device session table (one row per token instead of one `token_hash` column per user)? Currently no warning is shown to the kicked-out device — it just silently starts failing on its next authenticated action
- [ ] If multi-device support is wanted: add a sessions table (`user_id`, `token_hash`, `created_at`, `device_label?`), update `verifyToken` to check against it instead of `users.token_hash`, and decide whether/how to let users revoke old sessions

- New `/v1/vocab/word` endpoint sits behind the same Bearer auth middleware as all `/v1/*` routes; it validates the record shape and only touches the caller's own row. No new attack surface.
