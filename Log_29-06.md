## App Overview

Smart Translation is an Electron desktop app for intermediate-to-advanced English learners (B2‚ÄìC1 CEFR). It helps users build vocabulary by saving words encountered while browsing or reading, then provides AI-powered definitions, spaced repetition review, fill-in-the-blank practice, and gamified learning.

### Backend (Cloudflare Worker)
- **URL:** `https://smart-translation-api.fukumakino613.workers.dev`
- **Runtime:** Cloudflare Workers (Edge, no Node.js)
- **Router:** Hono
- **Database:** Cloudflare D1 (SQLite) ‚Äî users, auth_codes tables
- **Cache:** Cloudflare KV ‚Äî word analysis cache (7-day TTL), rate limits
- **AI:** Claude Haiku 4.5 via Anthropic API with prompt caching (‚â•1024 tokens)
- **Translation:** DeepL API (parallel with Claude, replaces Claude's translation field)
- **Email:** Resend API (`onboarding@resend.dev` sender for sandbox/dev)

### Auth Flow
Email OTP ‚Äî no passwords:
1. User enters email ‚Üí POST `/v1/auth/request` ‚Üí generates 6-digit code, hashes it (SHA-256), stores in D1, sends via Resend
2. User enters code ‚Üí POST `/v1/auth/verify` ‚Üí checks hash + expiry (10 min TTL), issues opaque bearer token
3. Token stored in `localStorage` (`cloud_auth_token`), sent as `Authorization: Bearer` on all subsequent requests

# Changes ‚Äî 29 June 2026

## Context
App was pushed to GitHub to generate DMG (Mac) and EXE (Windows) via GitHub Actions. Windows users reported the popup analysis feature failing. Debugging session conducted to identify and fix the root causes.

---

## Bug 1 ‚Äî `ACCOUNT_ID` placeholder left in Cloudflare AI Gateway URL

**File:** `server/src/analyze.js`

**Problem:** The `ANTHROPIC_GATEWAY_URL` constant contained a literal `ACCOUNT_ID` placeholder that was never replaced with the real Cloudflare account ID. Requests to the AI Gateway were failing with `403: {"type": "forbidden", "message": "Request not allowed"}`.

**Why Mac worked but Windows didn't:** Mac users' previous word lookups were cached in Cloudflare KV (7-day TTL). The popup returned cached results without hitting the gateway. Windows users had no cache (fresh install), so every request hit the gateway and failed.

**Fix:** Replaced `ACCOUNT_ID` with the real Cloudflare account ID `eeebef75914ebf0dddf7c498417a4e41`.

```js
// Before
const ANTHROPIC_GATEWAY_URL = 'https://gateway.ai.cloudflare.com/v1/ACCOUNT_ID/smart-translation/anthropic/v1/messages';

// After
const ANTHROPIC_GATEWAY_URL = 'https://gateway.ai.cloudflare.com/v1/eeebef75914ebf0dddf7c498417a4e41/smart-translation/anthropic/v1/messages';
```

**Also required:** The `smart-translation` AI Gateway did not yet exist in Cloudflare dashboard. Created it manually at dash.cloudflare.com ‚Üí AI ‚Üí AI Gateway ‚Üí Create Gateway ‚Üí named `smart-translation`.

## Bug 3 ‚Äî Popup auth token returning null on Windows (ongoing)

**File:** `desktop/renderer/popup.js`

**Problem:** After fixing the gateway URL, Windows users still got `401 Unauthorized`. The popup gets its auth token via IPC from the main process (`window.electronAPI.getToken()`). On Windows, this IPC call appears to return `null`, so the popup sends requests with no `Authorization` header.

**Root cause investigation:**
- Confirmed the API itself works fine: direct PowerShell test with the real token returned correct data.
- Confirmed the token exists in `localStorage` on the Windows machine.
- `app.js` does call `window.electronAPI.setToken(token)` on startup (line 1401) to share the token with the main process ‚Äî so the code path exists.
- Something in the IPC chain is failing silently on Windows specifically.

**Fix applied (beta.3):** Added `localStorage` as a fallback if IPC returns null. All Electron windows in the same app share the same localStorage partition, so the popup can read the token directly.

```js
// Before
const token = await window.electronAPI.getToken();

// After
const ipcToken = await window.electronAPI.getToken();
const lsToken = localStorage.getItem('cloud_auth_token');
const token = ipcToken || lsToken;
console.log('[popup] ipcToken:', ipcToken, '| lsToken:', lsToken, '| using:', token);
```

**Status:** Fix applied in beta.3, but the 401 persists on Windows. Both IPC token and localStorage token appear to be null or not reaching the request correctly. Further debugging required via popup DevTools (beta.4 opens DevTools for the popup window automatically).

**Next step:** Review console output from beta.4 popup DevTools to see exact values of `ipcToken` and `lsToken` at runtime on Windows.

## Beta.5 ‚Äî Token piggybacked on IPC message (architecture decision)

**Files:** `desktop/main.js`, `desktop/preload-popup.js`, `desktop/renderer/popup.js`

**What changed:** Removed the separate `window.electronAPI.getToken()` call from the popup. Instead, the main process now passes the auth token alongside the word in the `analyze-word` IPC message. The popup receives both in a single event and stores the token in `_token` for the lifetime of that analysis.

```js
// main.js ‚Äî sends token with the word
popupWindow.webContents.send('analyze-word', word, _authToken);

// preload-popup.js ‚Äî forwards both args
onAnalyzeWord: (cb) => ipcRenderer.on('analyze-word', (_, word, token) => cb(word, token)),

// popup.js ‚Äî receives token, falls back to localStorage
window.electronAPI.onAnalyzeWord(async (word, token) => {
  const lsToken = localStorage.getItem('cloud_auth_token');
  _token = token || lsToken || null;
});
```

**Why this is better than the old two-call pattern:** The old approach called `getToken()` separately inside the handler, which could return null if the main process hadn't stored the token yet (race condition, especially on Windows). Passing the token alongside the word collapses two async operations into one atomic IPC event.

**Long-term verdict:** Keep this pattern for Tauri. Tauri's IPC is more reliable but the single-message approach is still cleaner. Remove the `lsToken` fallback once confirmed stable on Windows.

## CORS / Worker Headers ‚Äî Decision for Public Release

**Current CORS config (`server/src/index.js`):**

```js
origin: (origin) => {
  if (!origin) return '*';                           // Electron / Tauri / curl
  if (origin.startsWith('chrome-extension://')) return origin;
  if (origin.startsWith('moz-extension://'))   return origin;
  if (origin === 'http://localhost:3000')       return origin;
  return null; // block everything else
},
```

**Why `!origin ‚Üí '*'` is correct for desktop:** Electron and Tauri do not send an `Origin` header on fetch requests. They fall into the `!origin` branch, which is the standard behavior for native clients. No change needed here.

**Auth is the real security gate:** The Bearer token in `Authorization` is what protects the API. CORS only gates browser-originated requests; it has no effect on curl, Postman, or desktop apps.

**What needs to change when launching the web app (Phase 4):** One line ‚Äî add the production web domain before `return null`:

```js
if (origin === 'https://your-web-domain.com') return origin;
```

No other header changes are required. The existing `allowHeaders`, `allowMethods`, and `maxAge` config is already production-ready.
